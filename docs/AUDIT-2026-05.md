# Production-readiness audit — May 2026

Full-system pass triggered by the team-management bug (deactivate/remove failed
with "id required"). That bug had three root causes — an ID-type mismatch, a
serverless fire-and-forget send, and a missing onboarding flow — so this audit
hunted every sibling of those classes plus the usual production checklist.

## Fixed in this pass

### 1. Serverless "fire-and-forget" notifications (HIGH — silent data loss)
Several notifications were dispatched without `await` and got killed when the
serverless function froze on response. They were silently never sent.
| Site | Notification | Fix |
|---|---|---|
| `team` POST | admin invite email | `await` (shipped earlier) |
| `dashboard/orders` PATCH | order status emails, SMS, web-push | `after()` |
| `dashboard/reviews` PATCH | review-approved email | `await` |
| `dashboard/products` PATCH | **back-in-stock emails** (also marked `notified_at`, so drops were unrecoverable) | `after()` |
| `order-fulfillment` | low-stock admin push | `await` |
| `subscriptions` | dispatch email | `await` |
| `push-admin` | device fan-out | `Promise.allSettled` |

Pattern going forward: use `after()` (Next 15) for best-effort side effects so
they complete after the response without blocking it — never bare fire-and-forget.

### 2. ID-type mismatch (the team bug) — audited, contained
Every `Number()`/`parseInt()` on an id (~30 call sites) was cross-checked against
each table's primary-key type. **Only `admin_users` and `profiles` use UUID PKs;**
the team route was the only place coercing a UUID — now fixed. All other
coercions are on `bigint`/`integer` PKs and are correct.

### 3. Admin onboarding (HIGH — couldn't actually add admins)
Dashboard login is password-only with no self-signup, and a new `admin_users`
row gives no way to log in. Added a real flow: the invite now generates a
Supabase set-password link (`admin.generateLink`) emailed via Resend, landing on
a new `/dashboard/set-password` page (exempted in middleware + shell).
**Requires one config step — see below.**

### 4. Raw DB error leakage (MED) — 38 sites
Authenticated account/dashboard routes returned raw Supabase `error.message`
(schema/internal detail) to clients. Now log server-side + return a generic
message. (Public routes were already hardened in an earlier PR.)

## Verified OK (no action)
- **ID coercion** outside UUID tables — correct.
- **Client secret exposure** — none (`process.env` server-only secrets never reach client bundles).
- **`profiles` / `customer_addresses` RLS-no-policy advisories** — INFO only; these tables aren't queried with the anon key (all access is service-role), so no policy is correct.
- **Order placement emails** — `fulfillOrder` awaits `Promise.allSettled(tasks)`, so confirmation/admin-alert emails are reliable (verified live).
